============================================================
1. Сущность «Кот» (Cat)
============================================================

1.1. Кот в реальном мире

  Примеры:
    - Кот Барсик, 2 года, рыжий;
    - Кошка Мурка, 4 года, чёрно‑белая.

  Что у кота есть (характеристики, свойства):
    - имя;
    - возраст;
    - окрас (цвет шерсти);
    - текущий статус/состояние:
        * спит;
        * ест;
        * играет;
        * точит когти;
        * просто ходит по квартире.

  Что кот умеет делать (поведение, действия):
    - мяукать;
    - есть;
    - спать;
    - играть;
    - точить когти о диван;
    - прыгать на стол.

1.2. Как это выглядит как класс (ПОКА ТОЛЬКО СЛОВАМИ, БЕЗ КОДА)

  Название класса:
    - Cat

  Свойства будущего класса Cat (поля):
    - name — имя кота;
    - age — возраст;
    - color — окрас;
    - currentState — текущее состояние (спит, ест, играет и т.д.).

  Методы будущего класса Cat:
    - meow() — «кот мяукает» и меняет своё состояние;
    - eat() — «кот ест»;
    - sleep() — «кот спит»;
    - play() — «кот играет»;
    - scratchSofa() — «кот точит когти о диван».

  Важная мысль для студентов:
    - мы не пишем пока код, а учимся мыслить в терминах:
        «какие у сущности есть ДАННЫЕ» и
        «какие ДЕЙСТВИЯ она может выполнять».
*/

============================================================
2. Сущность «Автомобиль» (Car)
============================================================

2.1. Автомобиль в реальном мире

  Примеры:
    - Toyota Camry, чёрная, пробег 150 000 км, в баке 20 л топлива;
    - Tesla Model 3, белая, пробег 10 000 км, заряд батареи 70%.

  Что у автомобиля есть (характеристики, свойства):
    - марка/модель;
    - цвет;
    - текущий пробег;
    - количество топлива (или уровень заряда);
    - состояние двигателя:
        * заведён;
        * заглушен.

  Что автомобиль умеет делать (поведение, действия):
    - заводиться;
    - глушиться;
    - ехать указанное количество километров;
    - заправляться топливом;
    - показывать текущий пробег и остаток топлива.

2.2. Как это выглядит как класс (ПОКА ТОЛЬКО СЛОВАМИ, БЕЗ КОДА)

  Название класса:
    - Car

  Свойства будущего класса Car (поля):
    - model — модель;
    - color — цвет;
    - mileage — пробег;
    - fuelInLiters — топливо в баке;
    - engineOn — флаг, заведён двигатель или нет.

  Методы будущего класса Car:
    - startEngine() — завести двигатель;
    - stopEngine() — заглушить двигатель;
    - drive(kilometers) — проехать указанное количество километров,
      уменьшив запас топлива и увеличив пробег;
    - refuel(liters) — заправиться на заданное количество литров.

  Важная мысль:
    - один и тот же «чертёж» Car позволит создать много машин
      с разными значениями полей (цвет, пробег, остаток топлива).
*/

/*
============================================================
3. Сущность «Платёж» (Payment)
============================================================

3.1. Платёж в реальной платёжной системе

  Типичный объект, с которым работают разработчики бэкенда:

  Данные платежа:
    - идентификатор платежа;
    - клиент (кто платит);
    - счёт/карта, с которой списываем;
    - сумма;
    - валюта (для упрощения можно опустить или зафиксировать);
    - статус:
        * NEW        — только что создан;
        * PROCESSING — обрабатывается;
        * COMPLETED  — успешно завершён;
        * CANCELLED  — отменён;
    - дата/время создания (в демо можно не использовать).

  Логика вокруг платежа:
    - создать платёж (инициализировать данные);
    - обработать платёж (проверить лимиты, списать деньги, поменять статус);
    - отменить платёж (изменить статус, возможно, откатить операции);
    - рассчитать комиссию;
    - распечатать/вывести чек (краткую информацию).

3.2. Как это выглядит как класс (ПОКА ТОЛЬКО СЛОВАМИ, БЕЗ КОДА)

  Название класса:
    - Payment

  Свойства будущего класса Payment (поля):
    - id — идентификатор;
    - clientName (или clientId) — информация о клиенте;
    - accountNumber — номер счёта или карты;
    - amount — сумма платежа;
    - status — текущий статус (NEW / PROCESSING / COMPLETED / CANCELLED).

  Методы будущего класса Payment:
    - process() — обработать платёж (сменить статус на COMPLETED,
      вывести диагностическую информацию);
    - cancel() — отменить платёж (сменить статус на CANCELLED);
    - calculateCommission() — вернуть размер комиссии;
    - printReceipt() — вывести краткий текстовый «чек» по платёжной операции.

  Связь с опытом БД‑разработчика:
    - раньше у нас были:
        * таблица PAYMENTS;
        * процедура process_payment(...);
        * процедура cancel_payment(...);
        * функция calculate_commission(...).
    - в ООП всё это объединяется в один класс Payment, где:
        * поля описывают СТРУКТУРУ данных;
        * методы описывают ЛОГИКУ, работающую с конкретным объектом.
*/

/*
============================================================
       Четыре принципа ООП на наших примерах
============================================================

1) Инкапсуляция
   - Во всех трёх сущностях (Cat, Car, Payment) мы не хотим,
     чтобы внешний код произвольно менял внутреннее состояние.
   - Примеры проблем:
       * коту ставят возраст -100;
       * машине руками обнуляют пробег;
       * платежу напрямую меняют статус с COMPLETED на NEW.
   - Поэтому:
       * реальные поля будут закрыты (private),
       * к ним будет доступ только через методы (get/set, process, cancel и т.д.).

2) Наследование
   - Кот (Cat) может быть частным случаем более общего Животного (Animal);
   - Клиент (Client) и Сотрудник (Employee) — частные случаи Человека (Person);
   - В следующих демо мы создадим:
       * базовые классы Animal, Person;
       * классы‑наследники Cat, Client, Employee.

3) Полиморфизм
   - Мы сможем писать код «обрабатывать платёж Payment», не зная,
     картой это платёж или банковским переводом;
   - Создадим подклассы CardPayment и BankTransferPayment,
     которые по‑разному реализуют метод process(),
     но будут использоваться через общий тип Payment.

4) Абстракция
   - Для кота, автомобиля и платежа мы выбираем ТОЛЬКО те свойства и действия,
     которые важны для нашей задачи:
       * нам не важно, как реально устроен двигатель;
       * нам не важно, какие именно мышцы у кота;
       * нам не важно, как физически движутся деньги между банками;
   - Мы работаем с высокоуровневыми понятиями:
       * «кот спит/ест/играет»,
       * «машина едет/заправляется»,
       * «платёж создаётся/обрабатывается/отменяется».

*/
